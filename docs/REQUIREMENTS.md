## 1. Scope & Overview
mail-haven is a **secure, reliable, and user‑centric email platform** targeting power users and enterprises that demand end‑to‑end encryption, custom domain support, and seamless integration with calendars, contacts, and file‑sharing services. The product will be delivered as a **web‑based SaaS** with a **React** front‑end, **Node.js/Express** back‑end, and **MongoDB** data store. All communications must be encrypted in‑flight and at rest, and the system must comply with GDPR, CCPA, and relevant ISO/IEC 27001 controls.

## 2. Functional Requirements  

| ID | Requirement | Description | Priority |
|----|-------------|-------------|----------|
| **FR‑1** | **User Registration & Authentication** | Users can sign‑up, log‑in, and recover passwords using **OAuth 2.0** (Google, Microsoft, Apple) or native email/password flow. MFA (TOTP) is optional but recommended. | Must |
| **FR‑2** | **Account Management** | Authenticated users can edit profile, change password, enable/disable MFA, and delete their account (GDPR “right to be forgotten”). | Must |
| **FR‑3** | **Secure Email Composition** | Compose, reply, forward, and draft emails with **end‑to‑end encryption (E2EE)** using **OpenPGP.js** (or equivalent). Encryption keys are generated client‑side and never leave the browser. | Must |
| **FR‑4** | **Encrypted Email Transmission** | All SMTP/IMAP/POP3 traffic must be wrapped in **TLS 1.3**. The platform must also support **SMTP over STARTTLS** for outbound mail to external domains. | Must |
| **FR‑5** | **Custom Domain Support** | Organizations can add one or more custom domains, verify ownership via DNS TXT record, and send/receive mail using those domains. DNS MX records are auto‑generated by the platform. | Should |
| **FR‑6** | **Calendar Integration** | Users can link external calendars (Google Calendar, Microsoft Outlook) via OAuth and view, create, edit, and delete events from within mail‑haven. Calendar data is stored encrypted at rest. | Should |
| **FR‑7** | **Contact Management** | CRUD operations for contacts, groups, and tags. Contacts can be imported/exported in **vCard** format. All contact data is encrypted at rest. | Must |
| **FR‑8** | **Secure File Sharing** | Users can attach files (max 100 MiB per file, 1 GiB per email) and share them via **time‑limited, access‑controlled URLs**. Files are stored encrypted (AES‑256‑GCM) and scanned for malware on upload. | Must |
| **FR‑9** | **Search & Indexing** | Full‑text search across subject, body, and attachments with **role‑based access control** (users only see results they are authorized to view). Search latency ≤ 200 ms for 10 k results. | Should |
| **FR‑10** | **Inbox/Folder Management** | Standard IMAP‑like folder hierarchy (Inbox, Sent, Drafts, Spam, Trash, custom folders). Server‑side rules for auto‑filtering (e.g., move to folder, mark as read). | Must |
| **FR‑11** | **Push & Email Notifications** | Real‑time push notifications (WebSocket/Server‑Sent Events) for new mail, calendar events, and file‑share alerts. Optional email fallback. | Could |
| **FR‑12** | **Admin Console** | Super‑admin role can view system health, manage tenant quotas, audit logs, and revoke compromised keys. | Must |
| **FR‑13** | **API‑First Design** | All functionality is exposed via a **RESTful JSON API** (versioned, `/api/v1/…`). Public SDKs (Node, Python) will be generated from OpenAPI spec. | Must |
| **FR‑14** | **Compliance & Auditing** | Automatic generation of GDPR‑compatible audit logs (who accessed/modified which data, when). Logs are immutable and stored for at least 12 months. | Must |
| **FR‑15** | **Internationalization (i18n)** | UI and email templates support at least **English, Spanish, German, French**. Language can be selected per user. | Should |
